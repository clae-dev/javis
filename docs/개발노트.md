# 자비스 개발 노트

개인용 AI 비서 '자비스'를 어떻게 만들었는지 정리한 문서. "무엇을"보다 **"왜 그렇게 했는지"**와, 만들면서 실제로 부딪힌 함정을 같이 적는다. 나중에 다시 봤을 때 설계 판단을 복기할 수 있게 하는 게 목적이다.

---

## 0. 설계 우선순위

가장 먼저 못 박은 것: **신뢰성 > 기능 개수 > 기술적 화려함.**

매일 직접 쓸 물건이라 "가끔 안 되는" 순간 안 쓰게 된다. 그래서 모든 선택에서 "이게 안정적으로 도는가"를 먼저 봤다. 화려한 구조보다 **부품 수를 줄이고**, 외부 의존을 최소화하고, 실패해도 죽지 않게(graceful degradation) 만드는 쪽을 택했다.

---

## 1. 전체 아키텍처

```
브라우저 / 설치형 PWA            PC 상주 음성 데몬(voice_client)
      │  WebSocket / REST              │  마이크·스피커 직접 사용
      └───────────────┬───────────────┘
                      ▼
              FastAPI Gateway
                      │
              LangGraph Agent ── PostgreSQL + pgvector
                      │            (장기기억 / 대화 체크포인트 / 리마인더 / 감사로그)
                      ▼
   Tools: 시간·웹검색·기억·리마인더·캘린더·Gmail
                      ▲
              APScheduler (능동 알림)
```

핵심 데이터 흐름(한 번의 대화):

```
사용자 입력
  → analyze   (의도 + 감정 동시 판단, 프로필 로드)
  → retrieve  (필요할 때만 장기기억 검색)
  → agent     (LLM + 도구 바인딩, ReAct 루프)
  → tools     (도구 실행, 위험 작업은 확인)
  → reflect   (기억할 것만 추려 저장 + 프로필 갱신)
```

## 2. 기술 스택과 선택 이유

| 선택 | 이유 | 버린 대안 / trade-off |
|------|------|----------------------|
| **FastAPI + WebSocket** | LLM 토큰을 받는 즉시 흘려보내야 체감 속도가 산다. | REST는 응답 전체를 기다려서 느리게 *느껴진다*. |
| **LangGraph** | 분기·루프·조건부 실행이 필수. 상태를 노드가 주고받는 그래프. | LCEL(선형 파이프)은 if문이 코드 곳곳에 박혀 금방 스파게티. |
| **PostgreSQL + pgvector** | 관계형 + 벡터검색을 DB 하나로. 운영 부담 절반. | 별도 벡터DB(Chroma 등)는 띄울 게 늘어남. |
| **Docker Compose** | 로컬 PC 한 대에 전부. 클라우드 비용 0. | — |
| **OpenAI** | 채팅·임베딩·STT·TTS를 키 하나로. | 멀티프로바이더는 v1엔 과함. |

### 단기 기억은 왜 Redis가 아니라 체크포인터인가

처음 계획엔 Redis가 있었는데 뺐다. 현재 대화 맥락은 **LangGraph 체크포인터**가 thread 단위로 들고 있어서 별도 캐시가 필요 없었다. Redis를 넣으면 "쓰지도 않는 부품"이 되어 운영 포인트만 늘어난다. → **죽은 부품은 만들지 않는다.** 세션 요약 캐싱이 실제로 필요해지면 그때 넣기로.

## 3. LangGraph 에이전트 깊이 보기

### 3.1 State 설계

```python
class JarvisState(TypedDict):
    messages: Annotated[list[AnyMessage], add_messages]  # 대화 누적(리듀서가 병합)
    intent: Optional[Intent]          # chat | query | task
    mood: Optional[str]               # 사용자 감정 추정
    retrieved_context: list[str]      # 장기기억에서 끌어온 것
    user_profile: dict                # 이름 + 누적 프로필 요약
```

**왜 쪼갰나:**
- `intent`를 먼저 분류 → "지금 몇 시야?" 같은 단순 발화에 풀 컨텍스트를 안 쓴다. 잡담은 기억 검색을 건너뛰어 비용·지연 절감.
- `messages`에만 `add_messages` 리듀서가 붙는다. 나머지 키는 노드가 반환하면 덮어쓴다 — 이 차이를 알아야 `user_profile` 갱신 시 기존 값을 직접 병합해 넣는다.

### 3.2 노드별 역할

- **analyze** — 의도와 감정을 *한 번의 LLM 호출*로 같이 뽑는다(비용 절약). 동시에 사용자 프로필을 DB에서 로드해 state에 싣는다. 실패하면 안전하게 `task`로 폴백(도구 접근 허용).
- **retrieve** — 마지막 사용자 발화로 장기기억을 벡터 검색. 실패해도 빈 리스트로 진행.
- **agent** — 시스템 프롬프트(성격+프로필+감정+관련기억) + 히스토리를 LLM에 넣고, 도구를 바인딩해 호출. 응답 또는 도구 호출을 만든다.
- **tools** — 도구 실행. 외부 영향 작업은 확인 절차(아래 4장).
- **reflect** — 대화 끝에 기억할 것만 추려 저장 + 프로필 갱신. **절대 본 흐름을 막지 않게** 전부 try/except로 삼킨다.

### 3.3 그래프 흐름과 라우팅

```python
g.add_edge(START, "analyze")
g.add_conditional_edges("analyze", _route,        # chat이면 retrieve 건너뜀
                        {"agent": "agent", "retrieve": "retrieve"})
g.add_edge("retrieve", "agent")
g.add_conditional_edges("agent", tools_condition, # 도구 호출 있으면 tools, 없으면 reflect
                        {"tools": "tools", END: "reflect"})
g.add_edge("tools", "agent")    # ReAct 루프: 도구 결과 들고 다시 agent
g.add_edge("reflect", END)
```

`tools_condition`은 LangGraph 프리빌트. 마지막 메시지에 `tool_calls`가 있으면 `"tools"`, 없으면 `END`("__end__")를 돌려준다. 그래서 `{END: "reflect"}`로 매핑해 "도구 안 쓰면 반추로" 흐르게 했다.

## 4. 사람 확인 (Human-in-the-loop)

메일 전송·일정 생성처럼 **외부에 영향을 주는 작업**은 실행 전에 반드시 사용자에게 묻는다. 이걸 그래프 흐름의 일급 시민으로 다뤘다.

```python
WRITE_TOOLS = {"create_event", "send_email"}

async def execute_tools(state):
    calls = state["messages"][-1].tool_calls
    write_calls = [c for c in calls if c["name"] in WRITE_TOOLS]
    if write_calls:
        approved = interrupt({...})   # 여기서 그래프가 멈춘다
        if not approved:
            return {...취소 메시지...}
    # 실제 실행
```

**함정 1 — interrupt 후 재실행:** `interrupt()`로 멈췄다가 재개하면 그 노드는 **처음부터 다시 실행**된다. 만약 읽기 도구를 먼저 실행하고 중간에 interrupt를 걸면, 재개 시 읽기 도구가 또 실행된다. → **확인을 맨 위에서 배치로 한 번만** 걸어, 재개 시 중복 호출이 없게 했다.

**재개 메커니즘:** WebSocket이 `confirm_request`를 클라이언트에 보내고, 사용자의 `{approved: true/false}`를 받아 `Command(resume=approved)`로 그래프를 이어 실행한다.

## 5. 메모리 시스템

대부분의 챗봇이 어설픈 이유: 메모리가 없거나, 다 기억하려다 컨텍스트가 터지거나.

### 5.1 계층

1. **단기** — 현재 thread의 메시지(체크포인터가 보관).
2. **장기** — `memory_items` 테이블 + pgvector 벡터검색.
3. **프로필** — `user_profile` 싱글톤 한 줄 요약. 매 응답 프롬프트에 주입 → "나를 알아가는" 느낌의 뼈대.

### 5.2 반추(reflect) — 핵심 차별점

모든 대화를 그대로 저장하면 검색 노이즈가 폭증한다. 그래서 대화 끝에 LLM에게 **"기억할 가치 있는 것만 3~5개로 추려라"** 시키고, 중요도 ≥ 4만 저장한다. 특히 **감정·관계·중요한 사건**을 놓치지 않게 카테고리(emotion/relationship/event/…)로 분류한다.

새 사실이 나오면 기존 프로필 + 새 사실을 합쳐 프로필을 갱신한다. → 다음 대화부터 자비스가 알아서 챙긴다.

### 5.3 벡터검색

```python
select(MemoryItem).order_by(MemoryItem.embedding.l2_distance(vec)).limit(top_k)
```

pgvector의 `l2_distance`로 가까운 기억 top-k. 임베딩은 `text-embedding-3-small`(1536차원, 비용 효율).

## 6. 감정 레이어

"감정이 들어가게"를 세 군데에 녹였다:

1. **성격 프롬프트** — 따뜻하고 눈치 빠른 동료. 기쁘면 같이 신나고, 걱정되면 솔직히 걱정. 단, 과하지 않게.
2. **감정 인식** — `analyze`가 매 턴 사용자의 mood를 읽어(intent와 한 번에) 프롬프트에 넣는다. → 형식적 위로가 아니라 상황에 맞는 반응.
3. **감정 음성** — `gpt-4o-mini-tts`의 `instructions`로 말투·억양을 조절("따뜻하고 다정한 친구 같은 말투로…"). 이건 gpt-4o 계열 TTS에서만 먹는다.

실제 검증: *"발표 말아먹어서 속상해"* → mood `속상함` 감지 → 공감 + 격려 + 구체적 도움 제안. memory에 emotion(중요도5)/event(중요도4) 자동 저장.

## 7. 음성

### 7.1 서버측 (브라우저용)

- STT: `/voice/stt` — Whisper로 받아쓰기.
- TTS: `/voice/tts` — mp3(브라우저) 또는 wav(데스크톱) 반환.

### 7.2 PC 상주 음성 데몬 (`voice_client/`)

"자비스 부르면 알아서" 기능. 브라우저가 아니라 **OS 마이크 직접** 사용 → 보안 컨텍스트 제약 없음.

```
1. 항상 대기 (로컬 Vosk로 '자비스'만 감지 — 깨우기 전엔 음성 전송 안 함)
2. 깨우면 띵! → 명령 녹음 (Vosk로 발화 종료 감지)
3. 녹음 → /voice/stt (Whisper, 정확도) → 텍스트
4. WebSocket /ws/chat → 에이전트 처리 (기억·도구·확인)
5. /voice/tts(wav) → 재생 → 다시 대기
```

**왜 Vosk(깨우기) + Whisper(명령) 조합인가:** 깨우기는 항상 도는 거라 가볍고 로컬이어야 한다(Vosk 작은 모델, 오프라인, 무료). 반면 명령은 정확도가 중요하니 Whisper로. 둘의 장점만 취함.

**프라이버시:** 깨우는 말 감지는 전부 로컬. 깨운 뒤에만 음성을 서버로 보낸다.

**되먹임 방지:** 자기 TTS 소리를 마이크가 다시 잡지 않도록, 말한 직후 입력 버퍼를 비운다.

## 8. 실시간 통신 (WebSocket)

토큰·도구상태·확인요청·능동알림을 한 소켓으로 주고받는다.

```python
async for mode, chunk in graph.astream(payload, config,
                                       stream_mode=["messages", "updates"]):
    if mode == "messages":
        msg, meta = chunk
        if meta.get("langgraph_node") == "agent" and isinstance(msg, AIMessageChunk):
            send {"type":"token", ...}        # agent 노드 토큰만 흘림
    else:  # updates
        if "__interrupt__" in chunk:
            send {"type":"confirm_request", ...}; return  # 확인 대기
```

**함정 2 — stream_mode를 리스트로:** `["messages","updates"]`로 주면 한 스트림에서 토큰(messages) + 노드 업데이트·인터럽트(updates)를 모두 받는다.

**함정 3 — 토큰 필터링:** messages 모드는 *모든* LLM 호출의 토큰을 흘린다(분류·반추 포함). `meta["langgraph_node"] == "agent"`로 최종 응답 토큰만 골라 보낸다.

**함정 4 — 동시 송신 경합:** 토큰 스트리밍과 능동 알림이 같은 소켓에 동시에 쓸 수 있다. 연결마다 `asyncio.Lock`을 두고 모든 송신을 직렬화했다.

## 9. 능동 알림

`APScheduler`(AsyncIOScheduler)로:
- 1분마다 마감된 리마인더 점검 → 연결된 클라이언트로 push.
- 아침 8시 오늘 일정 브리핑.

`ConnectionManager`가 연결된 소켓을 들고 있다가 broadcast. 알림이 와도 사용자가 답하는 중이면 락으로 안전하게 끼어든다.

**함정 5 — APScheduler 버전:** 4.x는 API가 완전히 바뀌어 `AsyncIOScheduler` import가 깨진다. → `apscheduler>=3.10,<4`로 상한을 박았다. timezone은 문자열을 주면 내부에서 pytz로 변환하니 문자열 그대로 OK.

## 10. 견고성

매일 쓰는 물건이라 "안 죽는 것"에 공을 들였다.

- **중앙 LLM 팩토리(`app/llm.py`)** — 모델 인스턴스 캐싱 + `max_retries=3`, `timeout=60`을 한곳에서. 호출처는 `chat()`/`fast()`/`embeddings()`만 쓴다.
- **감사 로그(`audit_log`)** — 모든 도구 실행을 기록. "안 시킨 메일이 갔다" 같은 상황의 유일한 추적 단서.
- **영속 체크포인터** — `AsyncPostgresSaver`로 재시작 후에도 대화 맥락·보류 중인 확인이 살아남는다. **실패하면 자동으로 인메모리(MemorySaver)로 폴백** → 어떤 경우에도 부팅은 된다.
- **방어적 노드** — retrieve/reflect는 실패해도 응답을 막지 않게 예외를 삼킨다.

**함정 6 — 두 개의 Postgres 드라이버:** SQLAlchemy는 `asyncpg`, LangGraph 체크포인터는 `psycopg`를 쓴다. `DATABASE_URL`에서 `+asyncpg`를 떼어 psycopg용 DSN을 따로 만들었다.

```python
@property
def psycopg_dsn(self):
    return self.database_url.replace("+asyncpg", "")
```

## 11. 배포 / 운영

- **Docker Compose** — `postgres`(pgvector 이미지) + `app`. healthcheck로 postgres가 healthy해진 뒤 app이 뜨게(`depends_on: condition: service_healthy`).
- **핫리로드** — `./app:/code/app` 볼륨 마운트 + `uvicorn --reload`. 코드 고치면 컨테이너 재빌드 없이 반영(파이썬 의존성 안 바뀔 때).
- **그래프는 lifespan에서 조립** — 체크포인터가 앱 수명 동안 살아있는 커넥션을 요구해서, import 시점이 아니라 `AsyncExitStack`으로 lifespan에서 빌드해 `runtime.graph`에 보관.

**함정 7 — env_file 변경:** `.env`를 고쳐도 실행 중 컨테이너엔 안 들어간다. `docker compose up -d`로 컨테이너를 재생성해야 새 환경변수가 반영된다.

**함정 8 — 서비스워커 캐시:** PWA의 서비스워커가 정적 파일을 캐시-우선으로 잡는다. `app.js`를 고쳐도 안 받아진다 → `sw.js`의 캐시 버전(`javis-shell-v1`→`v2`)을 올려야 새로 받는다.

**함정 9 — 브라우저 마이크:** `getUserMedia`는 **localhost 또는 https(보안 컨텍스트)** 에서만 동작. 폰에서 IP로 접속하면 막힌다. → PC 네이티브 음성 데몬으로 이 제약 자체를 우회.

### 검증 방법

```bash
docker compose up -d
curl localhost:8000/health                      # {"status":"ok", "openai":true}
docker compose exec postgres psql -U jarvis -d jarvis -c "\dt"   # 테이블 확인
docker compose logs app                          # "checkpointer: postgres" 확인
```

## 부록: 한 줄 교훈

- 부품을 줄여라. 안 쓰는 인프라는 만들지 마라.
- 실패해도 죽지 않게. 곁가지(기억·반추)는 본 흐름을 막지 않게.
- 외부 영향 작업은 무조건 확인을 거쳐라.
- "느리다"는 체감의 문제다 — 스트리밍으로 첫 글자를 빨리 보여줘라.
- 함정은 대부분 *경계*에 있다: 드라이버 경계, 캐시 경계, 보안 컨텍스트 경계, 재실행 경계.
